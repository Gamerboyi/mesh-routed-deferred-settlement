The Payment Journey
From a basement with no signal to a settled transaction in 4 steps
1
📱

Compose & Encrypt

Sender's phone builds a payment with a unique nonce, encrypts it using the server's RSA public key (hybrid RSA-OAEP + AES-256-GCM), and wraps it into a mesh packet with TTL=5.

2
🔄

Gossip & Forward

The packet spreads via epidemic/gossip protocol — every nearby phone broadcasts to neighbors. TTL decrements per hop. No phone can read the encrypted contents.

3
📡

Bridge Uploads

A phone holding the packet walks into 4G range and uploads to the backend. Multiple bridges may upload the same packet simultaneously — the "duplicate storm."

4
✅

Settle Exactly Once

Backend hashes → claims idempotency → decrypts → checks freshness → settles. Even 100 duplicate uploads result in exactly one debit/credit.

🔒
Three Hard Problems, Solved
Critical distributed systems challenges addressed end-to-end
Threat

🕵️ Untrusted Intermediaries

Strangers' phones carry the transaction. They could read amounts or tamper with the receiver.

Solution

Hybrid Authenticated Encryption. Payload encrypted with a fresh AES-256 key, that key wrapped with RSA-OAEP. GCM mode ensures any bit-flip causes decryption failure.

Threat

⚡ Duplicate Storm

3 bridge nodes hold the same packet. They all POST simultaneously. Naive processing drains the sender 3×.

Solution

Atomic Compare-and-Swap. SHA-256 hash of ciphertext as idempotency key. putIfAbsent() is atomic — exactly one thread proceeds, rest short-circuit in microseconds.

Threat

🔁 Replay Attacks

Attacker captures a valid encrypted packet and replays it months later to re-trigger settlement.

Solution

Nonces & Freshness Windows. Each payload has a UUID nonce and signedAt timestamp. Server rejects packets older than 24 hours. Identical payments produce different ciphertexts via unique nonces.

⚙️ Server-Side Ingestion Pipeline

Every packet from a bridge node passes through these 5 atomic stages in BridgeIngestionService

[1]
Hash Ciphertext

SHA-256 of raw ciphertext becomes the idempotency key

[2]
Claim Hash

Atomic putIfAbsent (CAS). Duplicates dropped instantly

[3]
Decrypt

RSA-OAEP unwraps AES key → AES-GCM decrypts + verifies auth tag

[4]
Freshness Check

signedAt must be within 24h. Rejects stale or future-dated packets

[5]
Settle

@Transactional: debit sender, credit receiver, write to ledger

🔐 Encryption Scheme

AsymmetricRSA-2048 with OAEP-SHA256 padding
SymmetricAES-256-GCM (authenticated encryption)
Key ExchangeFresh AES key per packet, wrapped by RSA
Tamper DetectionGCM auth tag — any bit-flip = exception
NonceUUID per payment (prevents false dedup)
